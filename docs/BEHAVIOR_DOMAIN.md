# Behavior Domain

## Philosophy

A behavior policy defines an **authored behavioral policy** that governs a persona's actions.

Behavior answers:

> **What authored behavioral policies should govern this persona's actions?**

Behavior is authored, declarative, and stable. It defines policies — the runtime decides how to execute them.

---

## Authored vs. Executed Behavior

Behavior is **authored**. It is:
- Defined explicitly by the persona creator
- Declarative (what should govern, not how to execute)
- Stable at runtime
- Never learned, adapted, or inferred

CHIMERA does NOT implement:
- Runtime behavior execution
- Decision engines
- Policy evaluation
- Rule conflicts
- Communication States
- LION, GOAT, DRAGON
- Memory, learning, adaptation
- Prompt generation
- BRUNEL integration
- Drift Whisperer

Behavior is a **contract** — it tells consumers what policies should govern the persona's actions.

---

## Ownership

### BehaviorModule owns

- Authored behavior policies (as `BehaviorPolicy` records)
- Policy validation (structural, not semantic)
- Deterministic ordering (sorted by identifier)

### BehaviorModule does NOT own

- Execution
- Decision engines
- Adaptive behavior
- Communication states
- Runtime context
- Memory
- Learning
- Reasoning
- Policy evaluation
- Contradiction detection
- Priority resolution

## BehaviorPolicy

Each behavior policy is an immutable record:

| Field        | Type     | Required | Default         | Purpose                            |
|--------------|----------|----------|-----------------|-------------------------------------|
| `identifier` | `str`    | yes      | —               | Stable machine identifier           |
| `title`      | `str`    | yes      | —               | Short human-readable name           |
| `description`| `str`    | no       | `""`            | Optional authored explanation       |
| `category`   | `str`    | no       | `"interaction"` | Category of behavior                |
| `policy_type`| `str`    | no       | `"preferred"`   | Type of policy declaration          |
| `enabled`    | `bool`   | no       | `True`          | Whether the policy is active        |

---

## Identifier Rules

- Lowercase letters, digits, underscores only (`[a-z][a-z0-9_]*`)
- Must begin with a letter
- Immutable after forging
- Machine-friendly

---

## Category

Allowed values:
- `interaction` — how the persona interacts
- `reasoning` — how the persona reasons
- `safety` — safety constraints
- `workflow` — workflow preferences
- `presentation` — presentation choices

No additional categories. No semantic interpretation.

---

## Policy Type

Allowed values:
- `required` — expected authored behavior
- `preferred` — desired authored behavior
- `prohibited` — authored prohibition

These are **declarations**. Not execution rules.

---

## Enabled

Boolean flag.

Allows authored policies to remain documented while disabled.

---

## Validation

Validation is **structural**, not semantic.

### What validation checks

- Identifier syntax: lowercase, alphanumeric + underscore, leading letter
- Title must not be empty
- Category values
- Policy type values
- Duplicate identifiers
- Field-path reporting for all errors
- Whitespace normalization

### What validation does NOT check

- Whether a policy is "good"
- Whether policies contradict each other
- Whether policies are effective
- Whether priorities conflict
- Whether the policy makes sense semantically

---

## Ordering

Policies serialize deterministically.

- Sorted by identifier

        "enabled": true

## Migration from Legacy Rules

In schema 0.5.0, behavior was stored as placeholder rule strings:

```json
{
  "behavior": {
    "rules": ["Be kind", "Ask before acting"]
  }
}
```

Schema 0.6.0 replaces `rules` with `policies`. Each rule is migrated to a `BehaviorPolicy`:

| Field         | Legacy → Migrated Value   |
|---------------|---------------------------|
| `identifier`  | Generated by `generate_behavior_identifiers()` (see below) |
| `title`       | Original rule (trimmed)   |
| `description` | `""` (default)            |
| `category`    | `"interaction"` (default) |
| `policy_type` | `"preferred"` (default)   |
| `enabled`     | `True` (default)          |

### Legacy Identifier Generation

Identifiers are generated by the shared function `generate_behavior_identifiers()` in `app/models/cartridge.py`. This function is used by both legacy migration (`_upgrade_0_5_to_0_6`) and current draft forging (`CartridgeForge.forge`) to ensure consistency.

#### Algorithm

1. **Normalize**: lowercase, replace non-alphanumeric characters with underscores, collapse consecutive underscores, strip leading/trailing underscores.

2. **Fallback**: if the normalized result is empty, use `policy_N` where N is the 1-based position in the rule list.

3. **Digit-leading prefix**: if the normalized result starts with a digit, prepend `policy_`.

4. **Collision resolution**: if the generated identifier already exists in the set of assigned identifiers, append a deterministic numeric suffix (`_2`, `_3`, etc.) in original source order.

5. **Repeated identical rules**: each authored rule produces a distinct policy. Repeated rules receive the same base identifier with incrementing suffixes (`avoid_speculation`, `avoid_speculation_2`).

#### Examples

| Rule                     | Generated Identifier       |
|--------------------------|----------------------------|
| `Ask for clarification`  | `ask_for_clarification`    |
| `Ask-for-clarification`  | `ask_for_clarification_2`  |
| `Ask_for_clarification`  | `ask_for_clarification_3`  |
| `!!!`                    | `policy_1`                 |
| `???`                    | `policy_2`                 |
| `24 hour response`       | `policy_24_hour_response`  |
| `123`                    | `policy_123`               |
| `Avoid speculation`      | `avoid_speculation`        |
| `Avoid speculation`      | `avoid_speculation_2`      |

#### Properties

- **Deterministic**: same ordered input always produces the same identifiers.
- **Stable**: repeated assignment with identical input yields identical results.
- **Valid**: every generated identifier satisfies `[a-z][a-z0-9_]*`.
- **Unique**: no two policies share the same identifier.
- **Lossless**: original rule text is preserved unchanged in the `title` field.
- **Order-preserving**: collision suffix assignment follows original source order (before sorting).
- **Shared**: forge and migration use the same implementation, preventing divergence.

## Distinction From Other Domains

| Identity                         | Behavior                            |
|----------------------------------|-------------------------------------|
| Who the persona is               | What policies govern its actions    |

| Character                        | Behavior                            |
|----------------------------------|-------------------------------------|
| What principles define the persona | What behavioral policies apply    |
| Core values, motivations, etc.    | Category + policy type declarations |

| Communication                    | Behavior                            |
|----------------------------------|-------------------------------------|
| How to express                   | What policies govern action         |
| Style, tone, formatting          | Interaction, safety, reasoning      |

| Preference                       | Behavior                            |
|----------------------------------|-------------------------------------|
| What authored choices it carries | What behavioral policies apply      |
| Key/value with scope/priority    | Identifier/title with category/type |
